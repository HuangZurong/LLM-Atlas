## 1. 上下文工程的思维模型

在深入机制之前，先理解整体框架：

### 1.1 一个核心与两个轴

**一个核心（One Core）**：把上下文窗口看作**极其昂贵、约束极强、而且并不可靠的系统 RAM**。当多个来源争夺这块有限空间时，上下文工程要决定该加载*什么*、按*什么顺序*加载，以及赋予*什么优先级*。

**两个轴（Two Axes）**：
- **空间轴（Spatial Axis）**：如何在单次请求内部安排信息（Prefix Caching、Lost in the Middle、Sandwich Pattern）。
- **时间轴（Temporal Axis）**：上下文如何跨多轮对话和代理循环演化（见 `07_Dynamic_Context_Management_cn.md`）。

### 1.2 演进的三个阶段

演进的**三个阶段（Three Stages）**：
1. 静态组装 → 2. 动态预算 → 3. 代理式编排。

*本文聚焦空间轴，以及驱动所有预算决策的物理约束。*

这三个阶段不是抽象标签，而是上下文工程在真实系统中的典型成熟路径。

#### 1.2.1 阶段 1：静态组装（Static Assembly）

这是最早期、也最常见的做法。开发者手工把上下文拼出来：
- 固定 system prompt
- 固定 few-shot examples
- 加少量历史
- 再附上用户 query

它适合单轮问答、FAQ、简单分类等任务。  
优点是简单直接；缺点是所有请求基本共用同一套结构，缺乏弹性，也几乎没有预算意识。

#### 1.2.2 阶段 2：动态预算（Dynamic Budgeting）

到这一步，系统开始真正把上下文当作资源来管理。它会：
- 先计算 token
- 再给不同层分配预算
- 超预算时按优先级裁剪
- 显式为输出预留空间

这类系统通常会区分：
- 哪些内容必须保留
- 哪些内容可以压缩
- 哪些内容可以删除

这是从“把东西拼进去”转向“按预算进行资源调度”的关键转折点。

#### 1.2.3 阶段 3：代理式编排（Agentic Orchestration）

当系统进入多步工具调用、长时间任务执行或多代理协作场景时，上下文已经不再是一份静态文本，而成了一个运行中的状态系统。此时需要处理：
- 工具结果不断累积
- scratchpad 持续膨胀
- 多轮推理链不断延长
- checkpoint、压缩、上下文重置
- 多代理之间的上下文隔离

这一阶段的重点，已经不是“写一个好 prompt”，而是“持续维护一个可运行、可收缩、可观测的上下文状态”。

### 1.3 四步流水线

面向生产系统的**四步流水线（Four-Step Pipeline）**：
1. **加载（Load）** → 2. **预算与排序（Budget & Sort）** → 3. **压缩与降级（Compress & Degrade）** → 4. **组装与观测（Assemble & Observe）**。

这四步构成了生产级上下文管理最核心的工作流。

#### 1.3.1 第一步：加载（Load）

先把候选上下文都收集起来，但这时还没有决定哪些一定会进入最终 prompt。候选来源可能包括：
- system prompt
- few-shot examples
- 用户画像或长期记忆
- RAG 检索结果
- 最近对话历史
- 工具结果
- 当前 query

这一阶段的本质是：**建立候选池，而不是直接拼装最终输入**。

#### 1.3.2 第二步：预算与排序（Budget & Sort）

收集完候选内容后，系统开始计算 token 并按优先级排序。它需要回答：
- 哪些内容绝不能删
- 哪些内容可以缩短
- 哪些内容一旦超限应最先丢弃
- 输出需要预留多少空间

这一步解决的是“资源分配”问题。许多系统失败，不是因为不会压缩，而是因为一开始就没做优先级排序。

#### 1.3.3 第三步：压缩与降级（Compress & Degrade）

如果候选内容超出预算，就不能直接截断了事，而要按层做有策略的降级，例如：
- 旧历史做摘要
- RAG 从 top-5 缩成 top-2
- 原始工具结果改成精简摘要
- 低优先级上下文直接丢弃

这里“degrade”的意思不是粗暴删除，而是**按重要性和信息密度逐级退让**。

#### 1.3.4 第四步：组装与观测（Assemble & Observe）

最后才把保留下来的内容真正拼成发给模型的上下文。此时不仅要关注顺序，还要记录观测数据，例如：
- 每层用了多少 token
- 是否命中缓存
- 是否触发压缩
- 总延迟与总成本

这一步的重点是：**一边组装，一边让系统可解释、可排查、可优化**。

## 2. 上下文窗口究竟是什么

上下文窗口是单次 LLM 调用的**总 token 预算** — 它同时包含输入（prompt）和输出（completion）。模型在这次调用中“知道”的一切，都必须装进这个窗口里。

```
┌─────────────────────────────────────────────────────┐
│                  上下文窗口（例如 128K tokens）      │
│                                                     │
│  ← 输入 Tokens ─────────────────→ ← 输出 →         │
│  [System][Memory][RAG][History][Query]  [Response]  │
└─────────────────────────────────────────────────────┘
```

### 2.1 `System`：系统提示词

`System` 是整次调用中优先级最高的一层。它负责规定模型在这次任务中的基本角色、行为边界、规则约束和输出要求。

典型内容包括：
- 角色定义：`你是一名客服助手`
- 行为规则：`若信息不足，不要编造`
- 输出要求：`请用 JSON 返回`
- 安全边界：`不要泄露内部策略`

如果把整份上下文比作一台程序正在运行的内存，那么 `System` 就像最上层的运行规则和执行约束。后面的所有内容，理论上都应在它的约束下被理解和执行。

### 2.2 `Memory`：持续有效的背景事实

在上下文工程里，`Memory` 不是模型参数中的“记忆”，也不是泛指一切历史信息。更准确地说，它指的是：**在当前这次调用中，被显式注入上下文窗口、并且与用户或任务持续相关的已知事实**。

它通常来自外部存储，而不是直接来自本轮输入，例如：
- 用户画像：`用户对海鲜过敏`、`用户偏好正式语气`
- 任务事实：`该工单已升级到 L2 支持`、`上次退款失败的原因是银行卡过期`
- 历史摘要：`前 20 轮对话已确认：目的地东京、预算 8000、出发日期 10 月 1 日`

工程上，可以把三者这样区分：
- **History**：原始对话过程，强调“发生过什么”
- **Memory**：从历史中提炼出的稳定事实，强调“哪些事实仍然有效”
- **RAG**：为了回答这次问题而临时检索来的材料，强调“这次回答要参考什么”

也就是说，`Memory` 更像“持续成立的背景事实”，而 `RAG` 更像“为这次任务临时调来的资料”。

### 2.3 `RAG`：为当前问题临时取回的外部材料

`RAG` 指的是 Retrieval-Augmented Generation 这条链路最终塞进上下文窗口的那部分检索结果。它的特点不是“稳定”，而是“与本次问题临时相关”。

常见来源包括：
- 知识库切片
- 产品文档摘录
- 法规条文片段
- 代码仓中的相关函数或文件段落

例如，用户问“退款政策里 30 天后还能不能申请”，系统可能会临时检索出政策文档里关于退款时限的 2 段内容，并把它们作为 `RAG` 塞进本次上下文。

所以，`RAG` 解决的是“当前问题需要查什么资料”，而不是“这个用户长期具备哪些背景信息”。

### 2.4 `History`：最近发生过的对话过程

`History` 指的是对话到当前这一步为止，最近若干轮的消息过程。它的作用不是补充外部知识，而是让模型知道：
- 刚才讨论到哪里了
- 前面已经确认了什么
- 当前问题是在延续哪个上下文

例如：
- 用户上一轮说“不是北京，是上海”
- 模型这一轮就必须知道“目的地”已经被改写

在实际系统里，`History` 往往不会无限增长，而是：
- 最近几轮逐字保留
- 更早内容压缩成摘要
- 再老的内容移入 `Memory` 或外部存储

### 2.5 `Query`：当前这一轮真正要解决的问题

`Query` 是这次调用最直接的触发点。它通常就是用户当前这一轮输入的任务、问题或指令。

例如：
- `帮我总结这篇文章`
- `订单 AC-7823 现在到哪了`
- `根据下面代码修复 bug`

如果说 `System` 规定“怎么做”，`Memory` 和 `RAG` 提供“做事所需的背景”，那么 `Query` 决定的就是“这一轮到底要做什么”。

### 2.6 `Response`：与输入共享预算的输出空间

`Response` 是模型最终生成的回答、计划、代码或动作结果。它不是独立于上下文窗口之外的，而是与输入**共享同一份总预算**。

这意味着一个常被忽视的工程事实：
- 输入塞得越满
- 留给输出的空间就越少
- 模型越容易因为长度限制被截断

所以在做上下文工程时，`Response` 不是最后才临时考虑的东西，而是从一开始就必须预留空间的一部分。

与人类记忆不同，上下文窗口具有以下特点：
- **扁平（Flat）**：没有内建的层级或重要性加权。
- **短暂（Ephemeral）**：调用结束后即被丢弃。
- **昂贵（Expensive）**：每个 token 都有延迟、显存和计费成本。

## 3. KV Cache：为什么上下文很昂贵

上下文中的每个 token，都会在 Transformer 的每一层注意力层中生成一个**键值（Key-Value, KV）对**。这些 KV 对会存储在 GPU VRAM 中。

### 成本模型

```text
KV Cache Memory = 2 × num_layers × hidden_dim × num_tokens × precision_bytes
```

以 70B 模型为例（例如 Llama 3 70B：80 层、8192 hidden dim、FP16）：
- 每 1,000 tokens 大约占用 ~2.5 MB VRAM
- 一个 128K 的上下文窗口，每次请求会消耗**约 320 MB**
- 如果要以完整上下文同时服务 100 个并发用户，仅 KV cache 就需要**约 32 GB VRAM**

这种硬件约束会直接转化为 API 调用的成本上限。

### 每个 token 的三重成本

| 成本维度 | 影响 |
| :--- | :--- |
| **延迟（Latency）** | token 越多 → Time-to-First-Token (TTFT) 越慢 |
| **内存（Memory）** | KV 条目越多 → 每张 GPU 可同时承载的请求越少 |
| **计费（Billing）** | 输入 token 会被计费（通常约为输出单价的 50%） |

### PagedAttention

vLLM 的 PagedAttention 像操作系统管理虚拟内存那样管理 KV cache — 它会为逻辑序列分配非连续的物理内存块。这样做无需改变模型行为，就能提升 batch size 和 GPU 利用率。

## 4. 前缀缓存（Prefix Caching，KV Cache Reuse）

当多个请求共享相同前缀（system prompt + few-shot examples）时，该前缀的 KV cache 可以在请求之间**复用**。

```
请求 1：[System Prompt][Examples][User Query A]
              ↑ 已计算并缓存 ↑
请求 2：[System Prompt][Examples][User Query B]
              ↑ 直接复用 — 无需重新计算 ↑
```

### 供应商支持（截至 2025 年初 — 请以官方文档为准）

| 供应商 | 机制 | 折扣 |
| :--- | :--- | :--- |
| **OpenAI** | Automatic Prefix Caching | 缓存输入 token 约 ~50% 折扣 |
| **Anthropic** | `cache_control` 参数 | 缓存 token 约 ~90% 折扣 |
| **DeepSeek** | 自动（>32 token 前缀） | 缓存 token 约 ~90% 折扣 |
| **vLLM（自托管）** | `--enable-prefix-caching` 标志 | 免费（因为 GPU 是你自己的） |

### 4.1 前缀缓存的反模式：动态注入

一个常见却杀伤力极大的陷阱，是把时间戳、UUID、session ID 等动态变量注入本应静态的内容块中。

```text
❌ BAD: 每秒都会打破缓存
System: "You are a helpful assistant. Current time: 2025-03-20 10:00:01"
User:   "What is my task?"

✅ GOOD: 保留缓存
System: "You are a helpful assistant."
User:   "[Time: 2025-03-20 10:00:01] What is my task?"
```

**工程规则（Engineering Rule）**：把动态变量隔离到提示词的最后部分（用户消息层）。绝不要用按请求变化的数据污染静态的 `[System Prompt]` 或 `[Few-shot Examples]`。

```
✅ [System Prompt][Few-shot Examples][Retrieved Docs][User Query]
   ←──────── 稳定、可缓存 ────────→ ←── 动态 ──→

❌ [User Query][System Prompt][Few-shot Examples][Retrieved Docs]
   ← 动态 → ←──────── 每次都缓存未命中 ────────→
```

## 5. 注意力模式与“中部遗失”（Lost in the Middle）

研究（Liu et al., 2023）表明，LLM 对以下位置的注意力最强：
1. 上下文的**开头**（首因偏差，primacy bias）。
2. 上下文的**结尾**（近因偏差，recency bias）。
3. 中间位置的信息则经常被忽略，或导致幻觉。

```
注意力
强度
  ▲
  │█                                                    █
  │██                                                  ██
  │███                                              ████
  │████                                          ██████
  │█████████                            ████████████████
  └──────────────────────────────────────────────────────→
  开头                                                结尾
                    上下文位置
```

### 缓解方法

- **Sandwich Pattern（三明治模式）**：把关键指令同时放在提示词的**开头**和**结尾**。
- **Recency Placement（近端放置）**：把最相关的 RAG chunk 放在**结尾**，不要埋在中间。
- **NIAH Testing**：上线前，使用 Needle-in-a-Haystack 测试验证模型在不同上下文位置上的检索准确率。

### 5.1 这种问题是不是所有长度的上下文都有

原则上说，**位置偏置几乎一直存在**，但它并不是在所有长度下都同样严重。

更准确地说：
- 在较短上下文中，这种现象可能已经存在，但往往不明显
- 在较长上下文中，它更容易发展成真实的工程问题
- 上下文越长、任务越复杂、中间区域越深，U 型分布通常越明显

因此，不应把它理解成“超过某个长度才突然出现”的问题，而应理解为：
**模型天然更偏向关注开头和结尾，只是上下文越长，这种偏置越容易被放大。**

可以这样粗略理解：
- **1K 左右上下文**：中间区域较浅，很多简单任务里偏置未必严重到影响结果
- **10K 左右上下文**：如果关键事实被埋在中部，且周围噪声较多，U 型效应往往会变得明显
- **100K 级上下文**：中部区域常会成为最危险的信息埋没区，必须有意识地调整摆放位置

所以工程上真正该问的不是“会不会有 Lost in the Middle”，而是：
- 在你的目标上下文长度下，这个问题严重到什么程度
- 你的模型在中部位置到底还能不能稳定检索
- 你的关键内容是否被放在了坏位置

这也是为什么长上下文系统在上线前，应当用 NIAH 或其变体做位置敏感性测试，而不能只看模型标称窗口大小。

## 6. 有效窗口与标称窗口

仅仅因为一个模型*支持* 128K tokens，并不代表它能在 128K tokens 上*有效地推理*。

| 窗口类型 | 定义 |
| :--- | :--- |
| **标称窗口（Nominal Window）** | 模型在不报错的情况下可接受的最大 token 数 |
| **有效窗口（Effective Window）** | 模型能可靠检索并推理的范围 |

大多数模型在远未达到标称上限之前，性能就已开始可测地下降。对于复杂推理任务，有效窗口通常只有标称窗口的 60–80%。

**实践含义**：不要把上下文填到 95%。模型需要留出“呼吸空间”。

### 6.1 输出安全边际

一个天真实现中经常出现、但又极其关键的疏漏，是把输入塞满整个上下文，导致没有空间留给模型生成输出。  
如果上下文预算是 128K，而你的输入已经占到 127.9K，模型可能只输出 100 个 token 就会突然停止（`finish_reason="length"`）。

**工程规则（Engineering Rule）**：始终定义严格的 `Output Reserve`（例如 4,000 tokens），并且输入上下文组装器绝不能侵占它。
