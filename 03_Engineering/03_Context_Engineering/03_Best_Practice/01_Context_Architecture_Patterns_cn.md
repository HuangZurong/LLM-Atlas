# 上下文架构模式

*前置要求：[../01_Theory/07_Dynamic_Context_Management_cn.md](../01_Theory/07_Dynamic_Context_Management_cn.md)。*

---

## 1. 四种上下文架构

### 模式 1：无状态重组（Fresh Assembly）

每次调用都从头重建上下文。不携带历史。

**适用场景**：单轮问答、文档分析、批处理。  
**优点**：简单、可预测、没有状态管理开销。  
**缺点**：没有对话连续性。

### 模式 2：滑动窗口

逐字保留最近 N 轮。预算超出时丢弃最老轮次。

**适用场景**：短到中等长度对话（< 20 轮）、客服。  
**优点**：简单，最近上下文始终连贯。  
**缺点**：会突然失去长程上下文。

### 模式 3：摘要 + 最近内容

将旧历史压缩为滚动摘要，最近 N 轮逐字保留。

**适用场景**：长对话、编码助手、研究会话。  
**优点**：兼顾连贯性与长程上下文。  
**缺点**：摘要有损；需要额外的压缩步骤。

### 模式 4：外部记忆 + RAG

把所有历史存到外部（vector DB），每次按 query 检索相关轮次。

**适用场景**：多会话应用、企业级助手、个性化场景。  
**优点**：历史几乎无限，可注入高度相关上下文。  
**缺点**：有检索延迟，可能漏掉隐式上下文。

### 决策树（ASCII）

```
这是单轮请求吗？
├── YES → 模式 1：无状态
└── NO
    ├── 该对话会跨多个 session 吗？
    │   ├── YES → 模式 4：外部记忆 + RAG
    │   └── NO
    │       ├── 预期对话长度 < 20 轮？
    │       │   ├── YES → 模式 2：滑动窗口
    │       │   └── NO  → 模式 3：摘要 + 最近内容
```

## 2. 静态内容优先规则（Static-First Rule）

始终把静态、可缓存的内容放在动态内容之前。这样可以最大化前缀缓存命中率。

```
✅ CORRECT（缓存友好）:
[System Prompt — static]
[Few-shot Examples — static]
[Retrieved Docs — 每次 query 变化]
[Conversation History — 每轮变化]
[User Query — 每次调用都变化]
 ↑ 长可缓存前缀 ↑

❌ WRONG（打爆缓存）:
[User Query]
[Conversation History]
[System Prompt]
[Few-shot Examples]
 ↑ 每次调用都缓存未命中 ↑
```

**影响**：以 Anthropic 的 prompt caching 为例（缓存 token 约 90% 折扣），一个 1000-token system prompt 在 10,000 次请求中复用，按 $0.015/1K tokens 计，可节省约 $1.35。

## 3. 上下文卫生反模式

| 反模式 | 问题 | 修复方式 |
| :--- | :--- | :--- |
| **什么都注入，“以防万一”** | 浪费 token、稀释注意力、提高成本 | 只注入与当前 query 相关的内容 |
| **把上下文填到 95%+** | 没有输出空间；模型响应会被截断 | 为输出预留 20–30% |
| **把指令只放中间** | Lost in the Middle — 模型会忽略它们 | 使用三明治模式：开头和结尾都放指令 |
| **跳过 NIAH 测试** | 上线实际上会静默失败的长上下文功能 | 上线 >32K 上下文前，务必跑 NIAH |
| **所有任务都用同一种上下文结构** | 简单任务过配，复杂任务反而不足 | 按任务类型决定上下文规模（见 `01_Theory/03_Context_Window_Mechanics_cn.md`） |
| **忽略工具结果累积** | 代理循环在 20–30 步后耗尽上下文 | 主动修剪/摘要工具结果 |
| **注入原始工具输出** | JSON 大块内容浪费 token | 注入前先提取相关字段 |
| **没有压缩兜底** | 上下文溢出时直接硬失败 | 始终准备优雅降级路径 |

## 4. 生产检查清单

在部署任何涉及上下文管理的功能之前：

- [ ] 已明确上下文预算（而不是依赖模型默认值）
- [ ] 输出预留至少占总窗口的 20%
- [ ] 静态内容位于动态内容之前（已启用前缀缓存）
- [ ] 关键指令已应用三明治模式
- [ ] 已在目标上下文长度上跑过 NIAH 测试
- [ ] 为每一层定义了压缩策略
- [ ] 设置了压缩触发器（soft / hard / emergency thresholds）
- [ ] 在代理循环中处理了工具结果累积
- [ ] 测量了上下文组装延迟（目标：<100ms p95）
- [ ] 已按层记录每次请求的 token 用量
- [ ] 已为平均输入 token > 窗口 80% 设置告警
- [ ] 已测试优雅降级（例如检索失败时会发生什么？）
