# Secure Architecture Patterns for LLM Apps

*Prerequisite: [02_Advanced_Threat_Modeling.md](02_Advanced_Threat_Modeling.md).*

---

Securing an LLM application requires more than just protecting the model. You must design the entire system with security as a first-class citizen, following Zero Trust principles.

## 1. The Gateway Pattern

Instead of allowing direct model access, use a **Security Gateway** to intercept and inspect all traffic.

```
User → Security Gateway (WAF + Guardrails) → Inference Engine
```

**Responsibilities of the Gateway**:
- **Prompt Injection Detection**: Scans for "ignore previous instructions" and other attack patterns.
- **PII Redaction**: Masks sensitive data in requests and responses.
- **Rate Limiting**: Prevents Model DoS and cost-based attacks.
- **Quota Management**: Limits token usage per user/organization.

## 2. The Dual-LLM Pattern (Quarantine)

For handling untrusted data (e.g., summarizing a public website), use a two-step process.

1. **Quarantine LLM**: A small, fast model extracts facts from the untrusted data into a structured JSON format.
2. **Privileged LLM**: A larger model takes the safe JSON output and incorporates it into the final response.

**Benefit**: Direct instructions found in the untrusted data are treated as "data" by the first model, preventing them from hijacking the control flow of the second model.

## 3. Least Privilege Tool Execution

When an agent uses tools, follow the principle of least privilege.

- **Sandboxed Execution**: Run code-execution tools in isolated containers (e.g., gVisor, Firecracker) without network access.
- **Role-Based Tooling**: A "Researcher" agent can only access `search_web`, while a "Support" agent can only access `view_ticket`.
- **Approval Gates (HITL)**: Require human approval for tools with side effects (e.g., `delete_record`, `send_email`).

## 4. Input-Output Decoupling

Never echo raw user input back in the response.

**Vulnerable**: `You asked about {user_query}. Here is the answer: {answer}`
**Secure**: `Here is the analysis based on your query: {answer}`

**Why**: Prevents reflected prompt injection attacks where the attacker tries to trick the user into viewing malicious content generated by the LLM.

## 5. Defense-in-Depth Checklist

- [ ] **Infrastructure**: Encrypt all data in transit (mTLS) and at rest.
- [ ] **Model**: Use specific system prompts that explicitly warn against following instructions in user data.
- [ ] **Code**: Sanitize all tool parameters before execution.
- [ ] **Observability**: Log all security-relevant events (blocked injections, tripped guardrails).
