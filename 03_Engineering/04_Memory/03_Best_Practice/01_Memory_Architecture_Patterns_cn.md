# 记忆架构模式：决策指南

*前置要求：[../01_Theory/01_Memory_Systems.md](../01_Theory/01_Memory_Systems.md)。*

---

选择正确的记忆架构，是 LLM 应用设计中影响最大的决策之一。本文基于真实生产场景，给出一个决策框架。

## 1. 决策树

```
对话是否为多轮？
├─ 否 → 不需要记忆。无状态 API 调用。
└─ 是
    ├─ 仅限单个会话？
    │  ├─ 短（<20 轮）→ 缓冲记忆（保留全部轮次）
    │  └─ 长（>20 轮）→ 混合记忆（摘要 + 滑动窗口）
    └─ 需要跨会话持久化？
        ├─ 结构化事实（偏好、决策）？
        │  └─ 实体记忆（JSON/图数据库）
        ├─ 非结构化召回（“我们上周讨论了什么”）？
        │  └─ 向量记忆（嵌入 + 检索）
        └─ 两者都要？
            └─ 分层架构（实体 + 向量 + 摘要）
```

## 2. 架构对比矩阵

| 架构 | 延迟 | 成本 | 召回质量 | 复杂度 | 最适合 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Buffer** | 最低 | 高（完整历史在上下文中） | 完美（窗口内） | 极低 | 短对话、原型验证 |
| **Sliding Window** | 低 | 中 | 良好（仅近期） | 低 | 客服机器人 |
| **Summary** | 中（LLM 调用） | 低 | 有损 | 中 | 长时运行会话 |
| **Hybrid (Summary + Window)** | 中 | 中 | 良好 | 中 | 生产级聊天机器人 |
| **Vector Retrieval** | 中（embedding + search） | 低 | 取决于 embedding 质量 | 高 | 跨会话召回 |
| **Entity** | 高（抽取型 LLM 调用） | 中 | 结构化、精确 | 高 | 个性化助手 |
| **Layered (All)** | 最高 | 最高 | 最佳 | 很高 | 企业级 AI 平台 |

## 3. 工业案例

### 3.1 ChatGPT Memory（OpenAI）
- **架构**：实体记忆（抽取后的事实）+ 用户可控 CRUD。
- **关键设计**：用户可以查看、编辑并删除单条记忆。
- **启示**：对于消费级产品，透明性与用户控制不可妥协。

### 3.2 Claude Projects（Anthropic）
- **架构**：项目级上下文注入（静态文档）。
- **关键设计**：不做动态记忆抽取；依赖用户整理好的上下文。
- **启示**：有时“笨”的静态上下文比“聪明”的抽取更可靠。

### 3.3 企业助手（内部工具）
- **架构**：分层（向量 + 实体 + 摘要）。
- **关键设计**：按用户、项目、组织分别维护记忆存储。
- **启示**：在企业场景中，多租户与访问控制至关重要。

## 4. 反模式

| 反模式 | 问题 | 修复方式 |
| :--- | :--- | :--- |
| **Infinite Buffer** | 上下文窗口溢出，成本爆炸 | 设置硬性 token 预算，并使用摘要压缩 |
| **Blind Retrieval** | 注入无关记忆，反而使模型困惑 | 使用相关性阈值（如 cosine > 0.8） |
| **No Deduplication** | 同一事实被存储 50 次 | 存储前做基于哈希或语义的去重 |
| **No Expiry** | 陈旧事实覆盖当前现实 | 时间戳 + 衰减评分 |
